## 一、Shadowrocket 的底层逻辑：你配置的每一条规则，都在和运营商斗智斗勇

很多用户以为，Shadowrocket 只是一个“开关”——打开就能翻，关上就能直连。错。大错特错。它本质上是一个**高性能的规则引擎**，其核心在于 **分流策略** 和 **协议栈优化**。

### 1.1 为什么你的“全局代理”会卡死？

我见过太多新人把 Shadowrocket 设置为“全局代理”。这在十年前或许可行，但在 2026 年，这无异于自杀。因为全局代理意味着你所有的流量（包括 DNS 查询、系统更新、甚至是一个小小的微信图片）都要经过那条远端线路。如果那条线路的带宽只有 100Mbps，而你的本地带宽是 1000Mbps，结果就是：**你的代理节点成了整个网络的瓶颈**。

**正确的做法是什么？** 使用 **配置式分流**。在 Shadowrocket 中，内置了极其强大的 GeoIP 和域名规则引擎。你需要做的，是让国内流量（如 `*.baidu.com`、`*.qq.com`）直连，让海外流量（如 `*.google.com`、`*.openai.com`）走代理。

**实操技巧：**
- 在 Shadowrocket 的“配置”页面，点击“编辑配置”。
- 找到 `[Rule]` 部分，确保优先级是：**最终规则（MATCH）** 必须放在最后一行。
- 添加类似 `DOMAIN-SUFFIX,googleapis.com,Proxy` 的规则，将特定的 API 域名强制走代理，避免因为 DNS 污染导致连接中断。

### 1.2 协议选择：别迷信“最强”，要相信“适配”

2026 年，V2Ray 的 VMess 协议已经进化到了 AEAD 加密时代，但很多用户发现，在移动网络（尤其是 4G/5G 的 NAT 环境下），**Trojan 协议** 的表现往往更加稳定。为什么？因为 Trojan 的流量特征更接近 HTTPS，运营商很难通过 DPI 进行直接干扰。

**但是，Shadowrocket 对 Trojan 的支持有一个致命陷阱：** 它默认的 TLS 指纹是 `chrome` 或 `safari`。如果你使用的是自定义的 CDN 回源（比如 Cloudflare），务必在 Shadowrocket 的节点配置中，将 `伪装域名` 和 `TLS 指纹` 手动设置为 `random` 或 `firefox`。否则，一旦 CDN 检测到你的 TLS 握手不符合浏览器标准，会直接阻断连接。

**深度建议：**
- **晚高峰首选：** Trojan + WebSocket + TLS（WS+TLS）。这是目前抗干扰能力最强的组合，尤其适合 Shadowrocket 在 iOS 上使用。
- **低延迟场景（如游戏）：** Shadowsocks + simple-obfs。虽然加密较弱，但握手延迟极低，适合对延迟极度敏感的 UDP 游戏。
- **千万别用：** 纯 Shadowsocks（无 obfs）直连。在 2026 年，这种流量几乎 100% 会被运营商在凌晨 2 点后随机限速。

## 二、线路的“生死抉择”：为什么你买的“专线”总在晚高峰失效？

这是最核心的问题。很多用户花了大价钱买了所谓的“IPLC 专线”或“IEPL 专线”，结果发现晚高峰依然卡成狗。原因很简单：**你买的可能只是“假专线”。**

### 2.1 真假专线的鉴别：从“延迟”和“抖动”入手

真正的 IPLC 专线，其延迟应该是**极低且恒定**的。比如从上海到香港，物理延迟大约在 30ms 左右。如果你买的“专线”在晚高峰延迟飙到 100ms 以上，或者出现大幅度的抖动（比如从 30ms 跳到 200ms 再跳回 30ms），那么恭喜你，你买到的很可能是一根 **“中转机”**——也就是普通的 BGP 线路，经过一个高防服务器中转。

**如何鉴别：**
1. **使用 Shadowrocket 的“延迟测试”功能。** 不要只看一次结果，要连续测试 10 次。如果每次结果差异超过 30ms，直接换线。
2. **检查路由追踪。** 在 Shadowrocket 中开启“日志”，查看连接建立时的 `tcp connect` 时间。如果这个时间超过 500ms，说明线路质量极差。
3. **终极手段：** 找一个靠谱的机场。我强烈建议你去 [高性价比机场精选大厅](/recommend) 看看。那里筛选出的机场，都是经过真实用户测试，且提供了 **晚高峰不限速、不丢包** 承诺的。那些在晚高峰还能保持 50ms 以内延迟的节点，才是真正的“钢铁专线”。

### 2.2 为什么要避开“共享带宽”的坑？

2026 年，很多机场为了压缩成本，会在同一台服务器上挂载几百个用户。即使你用的是 IPLC 专线，如果出口带宽只有 1Gbps，分摊到几百个用户手上，每个人能分到的带宽可能连 10Mbps 都不到。这就是为什么你的 4K 视频在白天能流畅播放，到了晚上就变成“幻灯片”。

**解决方案：**
- **选择“流量计费”的机场。** 那些按时间计费（比如月付）的机场，往往更倾向于超卖。而按流量计费（比如 100GB/月）的机场，因为成本是固定的，反而更愿意保证单用户带宽。
- **优先选择“中转 + 直连”的混合线路。** 好的机场会为每个用户分配一个独立的中转端口，确保你的流量不会和其他用户共享同一个出口。
- 如果你已经踩过坑，或者想彻底避坑，请务必阅读 [深入探秘储物戒防坑指南](/ring)，里面详细记录了如何通过“延迟曲线”和“丢包率”来识别那些割韭菜的“假专线”。这篇文章的每一句话，都是用真金白银换来的教训。

## 三、全平台客户端配置：从 Shadowrocket 到 Clash Verge 的终极协同

重度用户往往不止一台设备。你可能有 iPhone（用 Shadowrocket）、MacBook（用 Clash Verge）、Windows 台式机（用 Clash Verge Rev）。如何让这些客户端共享同一套配置，并且实现无缝切换？

### 3.1 配置文件的“大脑”：编写一份通用的 YAML 文件

Shadowrocket 支持导入 Clash 格式的配置文件（`.yaml`）。这意味着你只需要维护一份配置文件，就能在所有客户端上使用。

**关键配置项：**
```yaml
proxies:
  - name: "晚高峰专线-香港"
    type: trojan
    server: example.com
    port: 443
    password: "你的密码"
    udp: true
    sni: example.com
    alpn:
      - h2
      - http/1.1

proxy-groups:
  - name: "自动选择"
    type: url-test
    proxies:
      - "晚高峰专线-香港"
      - "晚高峰专线-新加坡"
    url: "http://www.gstatic.com/generate_204"
    interval: 300

rules:
  - DOMAIN-SUFFIX,google.com,自动选择
  - DOMAIN-SUFFIX,openai.com,自动选择
  - MATCH,自动选择
```
**注意：** 在 Shadowrocket 中，`url-test` 组的 `interval` 不能设置得太短（比如 30 秒），否则会频繁触发延迟测试，导致手机耗电剧增。建议设置为 300 秒（5 分钟）。

### 3.2 从 Shadowrocket 导出到 Clash Verge Rev

如果你使用的是 Mac 或 Windows，我强烈推荐你使用 [Clash Verge Rev 终极配置教程](/guides/clash-verge-rev)。它支持订阅转换、规则集订阅，以及最重要的——**内存占用极低**。

**操作步骤：**
1. 在 Shadowrocket 中，点击右上角的“复制”按钮，复制你的订阅链接。
2. 在 Clash Verge Rev 中，点击“订阅” -> “添加”，粘贴链接。
3. 在“转换”选项中，选择“保留原配置”或“使用远程配置”。**千万**不要选择“自动转换”，因为 Shadowrocket 的配置格式和 Clash 的配置格式在某些细节上（比如 `tls-fingerprint` 的写法）是有差异的。
4. 如果你不想折腾，也可以直接使用 [全平台客户端保姆级配置教程](/guides) 中提供的通用模板，它已经针对 Shadowrocket、Clash Verge Rev 和 Surfboard 进行了适配，直接导入即可。

## 四、AI 时代的“专线”新需求：如何让 ChatGPT 和 Claude 不卡顿？

2026 年，AI 访问已经成为重度用户的核心需求。但 AI 服务的流量特征和普通网页浏览完全不同：**它需要大量的 WebSocket 长连接，且对 DNS 解析的实时性要求极高。**

### 4.1 为什么你的 ChatGPT 总是“网络错误”？

很多用户发现，即使节点延迟很低，ChatGPT 也会频繁弹出“Network Error”。根本原因在于：**你的 DNS 解析被污染了。** ChatGPT 的 API 域名 `api.openai.com` 和 `chatgpt.com` 在国内的 DNS 服务器上，往往被解析到了错误的 IP 地址。

**解决方案（在 Shadowrocket 中）：**
1. 在“配置” -> “DNS” 中，将 DNS 服务器设置为 `1.1.1.1` 和 `8.8.8.8`，并开启“加密 DNS”（DoH）。
2. 在 `[Host]` 部分，手动添加：
   ```
   api.openai.com = 真实的 IP 地址
   chatgpt.com = 真实的 IP 地址
   ```
   这个 IP 地址可以通过在代理环境下 `ping api.openai.com` 获取。
3. **终极方案：** 使用 **分流规则**，将 `openai.com` 的所有流量强制走一个 **专门用于 AI 的节点**。这个节点必须是 **晚高峰稳定专线**，且支持 **UDP over TCP**（因为 WebSocket 的握手需要 UDP 支持）。你可以在 [晚高峰稳定专线推荐](/recommend) 中找到经过验证的 AI 专用节点。

### 4.2 流媒体解锁的“隐藏陷阱”

如果你想看 Netflix 的 4K 内容，或者使用 Disney+，单纯的节点延迟低是不够的。你需要的是 **原生 IP** 或 **解锁 IP**。很多机场提供的节点，虽然延迟低，但 IP 被 Netflix 标记为“数据中心 IP”，无法解锁。

**鉴别方法：**
- 在 Shadowrocket 中，开启“配置” -> “HTTP 请求” -> “UA 伪装”，设置为 `Mozilla/5.0`。
- 访问 `https://www.netflix.com/title/70143836`，如果看到的是“该内容在您所在地区不可用”，恭喜你，你的节点被识别为代理了。
- **解决方案：** 只选择那些明确标注了“流媒体解锁”的节点。好的机场会为每个节点分配不同的出口 IP，确保 Netflix、Disney+、HBO Max 都能畅快观看。

## 五、终极避坑指南：那些年我们交过的“智商税”

最后，我想和你分享一些血泪教训。

### 5.1 别迷信“无限流量”

2026 年，没有任何一家机场能提供真正的“无限流量”。所有声称“无限流量”的机场，要么是在后台偷偷限速（比如超过 10GB 后降到 1Mbps），要么就是跑路的前兆。**流量永远是稀缺资源。**

### 5.2 警惕“低价长期套餐”

那些 99 元/年、199 元/年的套餐，大概率用的是廉价的 CN2 GIA 或普通 BGP 线路。这些线路在白天或许可用，但在晚高峰（尤其是晚上 8 点到 11 点），丢包率可能高达 30% 以上。**一分钱一分货，在专线领域是铁律。**

### 5.3 永远保留一份“备用方案”

即使你选择了最好的机场，也难免遇到服务器维护、国际出口故障等意外情况。我的建议是：**始终准备一个备用节点**，可以是另一个机场的试用量，也可以是自己搭建的 VPS。在 Shadowrocket 的配置中，将备用节点放在 `fallback` 组里：

```yaml
proxy-groups:
  - name: "备用"
    type: fallback
    proxies:
      - "主节点"
      - "备用节点"
    url: "http://www.gstatic.com/generate_204"
    interval: 120
```

这样，一旦主节点不可用，Shadowrocket 会自动切换到备用节点，确保你的连接永不中断。
